## Executive Answer

### Top interpretation

The strongest current interpretation is:

- in the downloaded website app image, `0x08046528` and `0x08046544/48` belong
  to a metadata-like run, not to the live button-map / dispatch surfaces

That is not a claim that they are a perfect standalone `.ARM.exidx` section.
It is a narrower claim:

- they look much more like structured PREL31-style metadata than like live
  dispatch data

### Second-place interpretation

- some encoded dispatch representation is present, but it is incomplete without
  an additional decode/materialization step not present in the recovered
  `key_task` slice

### Lowest-probability interpretation

- these are the real live tables, and we are simply misreading ordinary handler
  data

The raw words argue strongly against that last one.

## 1. Strongest candidate interpretations

### Candidate A: metadata-like surface in the downloaded app image

Most likely.

This interpretation says:

- the website app image really does carry metadata-like bytes at these
  addresses
- raw code still references those addresses directly
- therefore the website app image is not a self-consistent standalone artifact
  for this runtime surface

This is currently the best fit for all evidence at once.

### Candidate B: encoded dispatch representation

Possible, but weaker.

This interpretation says:

- `0x08046528` is not a plain byte `btn_map`, but some encoded event surface
- `0x08046544/48` is not a plain handler table, but some encoded dispatch
  surface
- a missing decode step would have to exist somewhere between the raw loads and
  the final semantic owner call

The main problem with this interpretation is that the recovered `key_task`
dispatch path does not show the needed translation step.

### Candidate C: ordinary unwind metadata only

Directionally plausible, but too strong as stated.

The bytes behave enough like PREL31 metadata that the comparison is useful, but
the region is not yet cleanly proven to be one canonical unwind section with
all normal section-level properties.

So the current defensible phrasing is:

- unwind/extab-like metadata run

not:

- definitively solved `.ARM.exidx`

## 2. Evidence for and against metadata/unwind

### Evidence that argues **for** metadata/unwind-style interpretation

#### A. The corrected address model still lands on the same problematic bytes

With the corrected rule:

- project address = runtime literal - `0x4000`

the current Ghidra project and the raw app agree on the relevant bytes.

So this is not just a stale project-base mistake anymore.

#### B. `0x08046528` does not behave like a plausible byte event map

Representative bytes at runtime `0x08046528`:

```text
2F FE FF FF 72 03 00 00 DB F9 FF FF F6 0A 00 00 ...
```

That is not a credible plain logical-event lookup table.

#### C. `0x08046544/48` does not behave like a plausible direct handler table

The first 16 `key_task` slots are all impossible direct handler values:

```text
0:  FFFFF08B
1:  0000084D
2:  FFFFFB4B
3:  00000295
4:  FFFFFEB2
5:  0000009A
6:  FFFFFFAE
7:  FFFFFF95
8:  000000E4
9:  FFFFFE23
10: 0000038B
11: FFFFF9AF
12: 00000B44
13: FFFFE917
14: 00007058
15: 000029F4
```

None are plausible Thumb-call targets for `blx`.

#### D. The wider neighborhood decodes coherently under PREL31-style reading

Representative words from the same span decode to nearby code/metadata targets:

| Runtime word address | Raw word | PREL31 target |
|---:|---:|---:|
| `0x08046528` | `0xFFFFFE2F` | `0x08046357` |
| `0x0804652C` | `0x00000372` | `0x0804689E` |
| `0x08046544` | `0xFFFFF08B` | `0x080455CF` |
| `0x08046548` | `0x0000084D` | `0x08046D95` |
| `0x0804654C` | `0xFFFFFB4B` | `0x08046097` |
| `0x08046550` | `0x00000295` | `0x080467E5` |

That is substantially closer to metadata than to live dispatch data.

### Evidence that argues **against** a clean exidx/unwind conclusion

#### A. The region is being consumed as bytes and words by real code

The raw producer slices are real:

- button-scan code reads bytes from `0x08046528`
- `key_task` loads words from `0x08046544/48` and branches through them

Pure unwind data should not be consumed this way by application logic.

#### B. The ordering is metadata-like, but not a solved canonical unwind table

The decoded PREL31 targets show structure, but the neighborhood has not been
proven to be one clean, ordinary unwind section with all expected boundaries
and ordering properties.

So the evidence supports:

- metadata-like

more strongly than:

- definitely standard exidx

## 3. Evidence for and against encoded dispatch

### Evidence that argues **for** encoded dispatch

#### A. The code references are too direct to ignore

If the control-flow result is accepted, then stock code genuinely does treat:

- `0x08046528` as event-related byte data
- `0x08046544/48` as dispatch-related word data

That keeps an encoded-dispatch hypothesis alive.

#### B. The two surfaces sit close enough to support a shared packed scheme

A shared packed scheme is at least structurally imaginable:

- byte-oriented event side at `0x08046528`
- word-oriented dispatch side at `0x08046544/48`

### Evidence that argues **against** encoded dispatch

#### A. `key_task` shows no decode step before `blx`

The recovered sequence is:

```asm
add.w r0, r8, event, lsl #2
ldr.w r0, [r0, #-4]
blx   r0
```

If the loaded word is encoded, the decode step is missing from the recovered
slice. That is the biggest problem for this interpretation.

#### B. The loaded words are not close misses

The slot values are not "almost pointers." They are strongly unlike callable
targets.

That pushes the burden of proof onto any encoded-dispatch theory.

#### C. Both disputed surfaces share the same metadata cadence

The byte surface and the word surface both land inside the same structured
window. That is easier to explain as one metadata run than as two different
packed dispatch surfaces sharing one narrow range.

## 4. Current ranking

From strongest to weakest:

1. **Metadata-like run in the downloaded app image**
2. **Encoded dispatch representation with a missing decode/materialization step**
3. **Normal live button/disptach tables being misread**

## 5. Single best next forensic check

The single best falsification check is:

- obtain the exact internal-flash bytes from a programmed stock unit at runtime
  `0x08046528` and `0x08046544..0x08046580`, then compare them byte-for-byte
  against the downloaded website app image

Why this is the best check:

- it directly attacks the top interpretation
- it does not depend on more decompiler inference
- it does not depend on guessing more event semantics

How to interpret it:

### If the programmed stock bytes **differ**

That strongly confirms the current top interpretation:

- the downloaded app image is not carrying the live dispatch surface for this
  runtime window

### If the programmed stock bytes **match**

That quickly falsifies the top interpretation and forces a different conclusion:

- these metadata-like bytes really are the live runtime surface, so the missing
  piece must be a more exotic dispatch representation or a still-missed decode
  step

## Bottom Line

With control flow treated as settled, the data itself now points most strongly
to this conclusion:

- `0x08046528` and `0x08046544/48` are not behaving like live button/dispatch
  tables in the downloaded app image

The fastest way to break that conclusion is not another decompile pass. It is a
byte-for-byte comparison of this exact window against any programmed stock-unit
MCU image.
