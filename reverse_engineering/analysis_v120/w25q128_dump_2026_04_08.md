### Remaining Flash

Spot checks at `0x400000`, `0x800000`, and `0xF00000` were all `0xFF`.

Current working hypothesis:

- only the first `4 MiB` of the `16 MiB` part are programmed on this unit
- the remaining `12 MiB` appear erased

## Implications

### 1. External flash is real and readable

This confirms the bench firmware can stream the live W25Q128 contents over USB without opening the case.

### 2. The downloaded vendor app black-screen is not explained by a missing FAT volume

The board already has:

- a populated `System file` asset volume
- a second FAT volume for logo/screenshot paths

So the stock-app black screen is less likely to be caused by a completely absent external filesystem.

### 3. `9999.BIN` is still a live lead

The name exists, but the file is empty on this board. That leaves a few possibilities:

- stock creates or repopulates it dynamically
- the real factory image expected more content there on some units
- the firmware path that references `9999.bin` is optional or stale

The exact directory entry on this board is:

- attribute: `0x20`
- first cluster: `0`
- size: `0`

So this is not just an uninteresting tiny file; it is a true zero-cluster placeholder entry.

## `9999.BIN` Path Status In Firmware RE

The corrected V1.2.0 absolute string addresses are:

- `3:/System file/%d.jpg` at `0x080B9681`
- `3:/System file/9999.bin` at `0x080B9841`
- `2:/Screenshot simple file/%d.bin` at `0x080B9859`

Headless Ghidra xref tracing against those corrected addresses still returned no direct references in the current project (`flash_path_xrefs_2026_04_08.txt`).

Current interpretation:

- these path strings are probably consumed through an indirect resource/string table path
- or the relevant pointer/data references have not been reconstructed in the current Ghidra database
- so the old label "configuration file" remains plausible, but it is not yet grounded by a direct code xref

### 4. The external flash does not obviously hide a large opaque calibration blob

At least on this unit, there is no obvious extra filesystem or dense programmed region after `0x400000`. If missing scope state/calibration exists in external flash, it is more likely to be:

- a small structure inside the first two FAT volumes
- a file not yet extracted from the populated `System file` JPG-heavy volume
- or something the stock firmware synthesizes elsewhere rather than storing as a large raw blob
